### Auth 기본 개념
- 인증 VS 인가
    - 인증
        - 인증이란?
            - 너 누구야? -> 즉 로그인 과정 내가 누구인지 확인하는 것(로그인, 회원가입)
        - 요약
            - 내가 누구인가
    - 인가
        - 인가란?
            - 너 이거 할 수 있어? -> 즉 로그인 이후 권환 확인
        - 요약
            - 내가 뭘 할 수 있는가

---

- 전체적인 흐름
    1. 사용자가 id/pw 입력
    2. 서버가 사용자 확인 (DB 조회)
    3. 사용자가 맞다면 -> 토큰 or 세션 발급
    4. 클라이언트가 저장
    5. 이후 요청마다 인증 정보 요청

---


- 세션 VS 토큰
    - 세션
        - 세션이란?
            - 웹사이트나 애플리케이션에서 사용자가 접속한 시점부터 종료할 때까지의 연결 상태 및 상호작용 정보
        - 특징
            - 서버가 로그인 상태 저장
            - 클라이언트는 세션 id만 들고 있음
            - 서버가 상태 기억 -> 서버 부담 ^
        - 요약
            - 사용자가 로그인하면 서버가 기억하는 거임
    - 토큰
        - 토큰이란?
            - 사용자의 ID를 확인하거나 서비스 사용 권한을 부여하는 암호화된 데이터 문자열
        - 특징
            - 서버가 토큰 발급
            - 클라이언트가 직접 들고 다님
            - 서버는 상태 안 가짐
            - 확장성 좋음
        - 요약
            - 그냥 이제 서버가 토큰(즉 인증패 같은거라고 보면 됨)을 클라이언트에 발급하면 -> 클라이언트는 토큰을 가지고 알아 움직이는거

    정의
        - 인증과 인가를 통칭하는 it용어
        - 사용자의 신원을 확인하고 권한을 부여하는 보안 시스템
    
    인증
        - 클라이언트 측의 신원을 증명하는 것
        - 사용자의 정보를 보호하기 위함, 인증을 통해 클라이언트를 식별함으로써 추적 ㄱㄴ
        - 웹에서 ID, 비밀번호로 인증 및 식별함
    인가
        - 식별된 클라이언트가 특정 리소스에 접근할 수 있는 권한을 부여하는 것
    
    인증의 대표적 방식
        - 쿠키 방식
            - 사용자 인증에 대해 브라우저에서 사용자 정보를 저장하는 방식
        - 세션 방식
            - 사용자 인증에 대해 서버측에서 유저의 인증정보를 생성하고 클라이언트 측에서 인증번호 ID를 전달하여 쿠키가 전달하는 방식
            - 전통적인 stateful 방식으로 클라이언트의 상태를 서버에 계속 유지하고 해당 정보를 통해 인증하는 방식
        - 토큰 방식
            - 사용자 인증에 대해서 서버측에서 유저에게 토큰 발급, 클라이언트측 : 토큰 저장 -> 요청 -> 토큰을 서버에 전달
            - 기존 쿠키나 세션 방식에서 사용했던 쿠키를 사용하지 않음으로 기존 발생했던 취약점이 사라지고 stateless 상태로 상태를 유지하지 않아도 되기 때문에 확장성 증가
  ```
    대규모 웹 설계시 고려해야할 것
        - 가용성
            - 시스템이 지속가능한 상태인지(가동률)
        - 성능
            - 웹 사이트의 속도, 사용자 요청에 따른 응답 시간
        - 신뢰성
            - 어떤 자원에 대해 시스템이 항상 동일한 결과를 제공해야 하는 것
        - 확장성
            - 더 많은 부하를 처리하기 위해 처리량을 늘리는 것(트래픽 처리 + 저장공간 + 트랜직션 처리)
        - 비용
            - 서비스를 운영ㅇ하고 관리하는데 드는 비용
        - 관리성
            - 시스템을 얼마나 쉽게 운용하고 문제에 대한 분석이 이해하기 쉬운 편인가
    
    대표적인 토큰 기반 인증 방식
        1. OAuth2.0 인증
            - 신뢰할 수 없는 네트워크 환경에서 제 3자 서비스에게 인증정보를 넘기지 않고 하나의 아이디로 여러 서비스에서 사용가능하게 하는 sso 인증 프레임 워크
            - 흐름
                1. client가 resource owner에게 보호된 자원에 접근하기 위해 인증 요청을 전달한다.
                2. Resource Owner는 권한 부여 동의서를 발급한다(RFC의거 4가지 유형)
                3. 권한 부여 방식에 따라 Authorization server측에 인증 및 접근 토큰을 요청함
                4. Resource owner에게 접근 할 수 있는 범위 및 기간등의 정보가 담긴 접근 토큰을 제공
                5. client는 토큰을 통해 resource server에 보호된 자원을 요청
                6. resource server는 접근 토큰에 담긴 자원 접근 범위 및 유효기간을 확인하며 자원을 제공함
        2. JWT Token 인증 방식
            - 토큰 기반 인증 방식 중 하나로 사용자의 일부 자원 및 인증에 대한 서명 정보를 포함하여 Token 자체로 보호된 자원에 대해 요청하는 방식
            - 구조
                -aaaaaa(헤더).bbbbbb(내용).cccccc(서명)
                - 헤더 : 헤더, payload, signature 정보를 암호화 할 방식 (alg),(type)등 이 포함
                - 내용 : 인증한 사용자의 고유 ID 혹은 특정 리소스의 내용등을 의미 보통 각각의 리소스를 클레임이라 부름
                - 서명 : base64encode 한 header,payload 및 secret key를 포함한 뒤 alg 에 의한 hash 알고리즘을 이용하여 서명 한 갓ㅂ
            - 인증 흐름
                1. 클라이언트가 사용자의 ID/PW를 통해 인증 시도
                2. 서버측에서 사용자에 대한 식별을 한 뒤 secret key를 통해 JWT 토큰 발급
                3. 클라이언트는 JWT 토큰을 통해 서버의 보호된 자원(API)을 요청
                4. 서버는 자원 요청에 대해 토큰의 서명을 통해 확인 후 유효성 확인 후 보호된 자원 응답
            - 장점
                - JWT의 경우 발급한 뒤 토큰 검증만 하면 되기 때문에 저장소의 부하를 주지 않음
                - RESTFul API 친화적 및 토큰 관리에 대한 부담을 덜어줌
            - 단점
                - 한번 발급된 JWT 토큰의 경우 탈취된 경우 유효기간이 완료되기 전까지 사용가능
                - JWT 토큰의 길이가 길어지기 때ㅜㅁㄴ에 자원 요청에 대한 전반적인 네트워크 비용이 높아짐
                - Payload 정보가 base64로 인코딩 되어 있기 때문에 담을 수 있는 정보가 제한적
```
